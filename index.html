找工作进行时

发表于 2016-02-29   |   分类于 努力上进   |  

想起某篇文章的标题，大意是如果你想做成什么事，就别急着发朋友圈。或许这种如何努力，如何单调生活不适合在朋友圈上刷存在感，那么就淡定点，就让自己默默感受这个或许有点艰辛但回忆起来还是挺美好的过程吧。

9月份实习之后，开心地回了趟国，回美帝后就发现刷过的题都忘得差不多了，然后又赶上号称形势不那么好的一年，对于我这个转专业，基础又一般般的来说，确实不容乐观。狂刷两周面经，还是华丽丽地挂了几个公司。然后似乎便错过了最好的时间段。12月份基本上没有什么进展，做几个OA，然后要么没下文，要么处理得非常缓慢。特意找了级别比较高的人内推，还是收到了not moving forward的邮件。

还好年初的时候，很多公司又陆续开始校园招聘了，然后就发现有更多的公司在招intern。那就继续刷题，继续面试吧。我不是大牛，也没想过拿一拨offer，只是在lg的push下和鼓励下，拿到了比较满意的offer。

下面就是自己的一点小经验和小感悟吧。

Timeline

  • 找工作的timing真的很重要，对我来说，大概3月份左右开始找intern, 4月份拿到intern offer已经很晚了。其实1月份开始就可以积极投intern了，当然有些更牛的人可能是下半年就有实习offer了，所以说，早投真没什么坏处。
  • 而找full time更是越早越好，有些公司还有冷冻期。如果早点面，6个月后还没毕业还能再尝试一次，又有什么不好呢。况且的况且，如果只是通过网上海投，处理时间真的很难预料，尤其那些比较热门的大公司。full time的招人高峰期应该就在9-11月，春季的时候也有一些。
  • 很多人可能会担心自己还没准备好，不想贸然去投，但这样很可能就错过了最好的申请时间，影响可能更大。在面试过程中积累实战经验也是很有效的提高方法。

前期准备

简历

  • 简历可以多找朋友或是比自己有经验的人改改，多一个人看看总能多一些想法。比如可以加上一些显得fancy一点的描述，可以把一个普通的project写成实现了某某system。当然，加一些数据会显得更有说服力，而句子要尽量言简意赅。一些关键词，比如掌握了某种技术，某种framework，也可以一下子被注意到。
  • 改简历的过程其实也是加深自我认识的过程，当你对简历上每一条都清晰明了的时候，会有更多信心应对简历相关的任何问题。
  • 投简历的途径，首选便是找人内推。其次是参加学校CS的career fair，但那种general的career fair就不那么推荐了。我还没尝试过直接在linkedin上找recruiter，听说也是不错的方法。最其次的方法便是网申，当然大牛应该在哪都被人哄抢吧。

刷题

  • 对于转专业的我来说，多年前的C, C++已经忘得差不多了。相当于从0开始学Java, 上algorithm。 在leetcode上刷题确实帮助很大，一方面运气好的时候经常能碰到leetcode原题，同时，讨论区也总有一些很巧妙的解法可以好好学习，更重要的是这个过程培养了一些解决问题的思路。至少在碰到新题时，不会毫无头绪，会尝试将它转化成之前见过的问题。同时在这个过程中，也加深了解了某些看似基础的data structure和algorithm。
  • 不过，刷leetcode要适时做一些总结，否则很容易刷完后面的就忘了前面的。可以将相似的问题放在一起多做比较，时不时翻出来想想为什么这么做。

面经

当把leetcode刷熟练后，除了每天保持一定的刷题量，收效最大的就是看近期的面经了。一亩三分地上的面经版，很棒，有很多很新的面经，而且会发现有些题确实反复出现过。如果有时间，最好可以把自己不是完全有把握的题都写一遍，如果真正面试的时候碰到，就赚大了。同时，去论坛上看大家的面试经历是一种学习，更是一种激励，当发现大家都开始on site了，自己简历还没有消息，那就肯定要加倍努力了。

面试过程

知识储备

因为我都是申请general的SE职位，对于我面试过的公司来看，面试的问题基本上都是一些正常范围内data structure, algorithm, OO design, system design，CS基础的问题。我竟还没碰到过让写multi-thread，或是让处理concurrency，算比较幸运吧。

  • data structure: array, linkedlist, tree, graph, queue, stack, hashmap, hashset, heap， adjacency list, adjacency matrix等是最简单的，复杂点的比如Trie, segment tree, binary indexed tree, quad tree, surffix tree, AVL tree, RB tree等，主要前面三个leetcode都有用到，要会写。
  • algorithm，列几个感觉碰到比较多的：
    1. sorting: quick sort, merge sort, bucket sort
    2. searching: binary search
    3. dfs, bfs, toplogical sorting, union find
    4. 一些technique: divide conquer, dp, memorization, backtracking, recursion，两指针（如binary search时），左右两边受限的情况（如trapping rain water)
  • OO design: 了解一些基础的design pattern, 多看看一些典型题，比如电梯，parking lot，CC150上有一大章讲OO design，可以好好研究下并进行总结。
  • big data: 这篇博文 虽然名字浮夸了点，但基本上涵盖了很多大数据处理的想法，至少在碰到类似问题时会有一些思路。
  • system design: 从一些典型题入手，了解相类似的架构，其实简单来说都是load balancer加n个server加n个storage，然后具体到Cache怎么加，怎么实现high performance, availability等等，可以看这篇文章。
  • 其他：虽然可能某些职位的描述里需要有某特定方面的经验技能，但实际考查时也没有完全focus在这方面。当然还要看具体公司的风格，大公司一般上更注重CS的基础知道，以及解决问题的思维思路。当然如果简历上强调自己某项特别突出的skill，就要准备好应对了。

面试交流

  • 一般的IT公司面试流程都是非常专业的，面试官也都进行过相应的训练。他们会尽量让你放松，会适时引导你。我目前还没有碰到特别变态的面试官，最多就是脸上没什么表情，只是沉默地看着你一个人在那说或是写。所以尽量保持一个良好的心态，碰到难题，大家都一样，依然要努力尝试解决这个问题。
  • 面试过程中的交流很重要，不要害怕说错什么，要适时表达你在想什么。要不然面试官无法了解你的思路，也无从判断你是不是真的会做这道题。像design类的题更加需要多交流，讨论具体的设计需求才能保证没有偏离面试官希望的轨道。大部分时候要按照面试官引导的方面走，尽量不要跟面试官进行对抗。
  • 如果一道code题知道怎么做，一般就说一下自己的思路，看面试官是否同意，然后再开始写。如果不同意，就要再讨论是不是要改进复杂度，是不是有些情况没有很好的处理到。如果这道题完全不知道怎么做，一般面试官会给一些提示或是进行一些简化，然后大胆地把自己的想法说出来，想到什么说什么。一般比较好的面试过程，就是给一个相当复杂的问题，从简化的小问题开始，在引导下一步步解决。

面试结束

心理上坦然地move on，毕竟已经面完了，不要有太多懊悔或是兴奋，尽量让自己平静客观。如果发现有哪些知识点有欠缺或是准备的不完善，或是面试时交流有问题，都需要立刻提醒自己下次继续努力。面试的过程也是一个快速成长的过程。如果可能的话，尽量向recruiter要feedback，但我目前还没有成功拿到过。然后，就耐心等待或好或坏的消息，学会在这条路上更坚定地努力下去。
